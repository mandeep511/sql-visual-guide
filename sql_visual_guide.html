SQL — A Visual Guide

Everything here is explained step by step. No shortcuts, no assumptions. Read top to bottom.

The most important thing to know before you start

In every example below, you'll see names like lead_name, deal_value, leads, etc.

These are NOT fixed words. They are names that someone made up when they created the database. Your database will have completely different names — like customer_name, order_total, or products.

The only words that are fixed are the blue words — those are SQL commands (SELECT, FROM, WHERE, etc.). Everything else changes depending on your data.

SQL writing rules — the small stuff that trips everyone up
Names (tables & columns)
No spaces allowed. Use underscores.
first_name order_total
first name order total
Don't put quotes around names.
Text values
Always wrap in single quotes.
'Mumbai' 'West'
"Mumbai" Mumbai
SQL uses single quotes ', not double ".
Numbers
Just type the number. No quotes.
500000 90
'500000' "90"
No commas either — write 500000 not 500,000.
Equals sign
SQL uses a single = for "equals".
Not == (that's other languages).
WHERE city = 'Mumbai' is correct.
These rules apply everywhere in SQL. You'll use them in every single query you write. Come back to this box whenever you're unsure.

Correct! Column names use underscores instead of spaces, and don't need quotes around them.

Part 1
SELECT + FROM
Every SQL query starts with these two. Always. No exceptions.
The pattern (this is the shape — memorise this)
SELECT column_names_you_want FROM table_name;
What each word does
SELECT Tells SQL which columns to show you. Columns = the vertical headings in your data (like columns in Excel).
FROM Tells SQL which table to look in. A table = one sheet of data (like one tab in Excel).
* Star means "all columns". So SELECT * means "show me everything".
; Semicolon = full stop. It tells SQL "I'm done writing my query".
You have an Excel sheet. You want to see only some columns. You'd hide the rest. SELECT does exactly that — it picks which columns to display.
SELECT + FROM is the only required pair. Every other keyword you'll learn (WHERE, ORDER BY, LIMIT, etc.) is optional — you add them only when you need them.
Things SQL does NOT care about
Upper or lower case
SELECT = select = Select
All the same to SQL. We use UPPERCASE for keywords by habit — it makes them easier to spot. But it's not required.
Line breaks & spacing
You can write a query on one line or spread across ten.
SQL ignores extra spaces and line breaks. We use multiple lines to make it readable for humans.
Things SQL DOES care about
Commas between columns
SELECT name, age, city
SELECT name age city
Commas separate column names. No comma after the last column.
Exact column names
If the column is called first_name, you must write exactly that.
firstname first name First_Name
Typos = error. Check the schema for exact spelling.
Example with sample data
SELECT lead_name, region, deal_value FROM leads;
What SQL returns
lead_nameregiondeal_value
Riya MehtaWest800,000
Arjun NairNorth1,500,000
Priya ShahSouth650,000
Karan BoseWest1,200,000
The columns appear in the order you listed them. If you wrote SELECT region, lead_name, region would be the first column. You control the layout.
Same idea, different names
SELECT product_name, price FROM products;
See? Different table, different columns. The pattern is the same. Only the green names changed.

You want to see first_name, email, and city from a customers table. Which query is correct?
Column names need commas between them! Without commas, SQL doesn't know where one name ends and the next begins.
Correct! Commas separate each column name. No comma after the last column (city) — the next word is the keyword FROM.

Part 2
WHERE
Filters your data. Only keeps rows that match a condition you set.
The pattern
SELECT columns FROM table WHERE column operator value;
What each word does
WHERE Starts the filter. Only rows that pass this test will show up.
AND Add more conditions. ALL of them must be true.
OR ANY one condition being true is enough.
= > < >= <= != Comparison operators. = means "equals", != means "not equal to", > means "greater than".
'text' Text values must be wrapped in single quotes. Numbers don't need quotes.
Excel's Filter button. Click a column, pick a value, and only matching rows stay visible. WHERE does the same thing — but you type the filter instead of clicking it.
WHERE is optional. If you leave it out, you get ALL rows. You only add WHERE when you want to narrow things down.
Comparison operators — what each symbol means
=equalsWHERE city = 'Mumbai'
!=not equal toWHERE status != 'cancelled'
>greater thanWHERE price > 100
<less thanWHERE age < 30
>=greater than or equal toWHERE score >= 90
<=less than or equal toWHERE salary <= 50000
= means "equals" in SQL. Don't use == — that's from other programming languages. In SQL, one = is enough.
Text on the right side always needs single quotes. Numbers never do. = 'West' vs > 500000.
Filtering different types of data

The right side of the comparison changes depending on what kind of data the column holds:

Filtering text — always use single quotes
WHERE region = 'West' -- text value in single quotes WHERE status = 'completed' -- another text value WHERE city != 'Mumbai' -- not equal to Mumbai
Filtering numbers — no quotes
WHERE deal_value > 500000 -- number, no quotes, no commas WHERE age >= 18 -- another number WHERE quantity = 0 -- zero is still a number
Filtering dates — single quotes (they look like text to SQL)
WHERE order_date > '2024-01-01' -- after Jan 1, 2024 WHERE hire_date = '2023-06-15' -- exact date match
Dates are written in 'YYYY-MM-DD' format (year-month-day) with single quotes. '2024-01-15' means January 15, 2024.
Example — putting it together
SELECT lead_name, deal_value FROM leads WHERE region = 'West' AND deal_value > 500000;
How SQL processes each row

Priya ShahSouth650,000dropped — South ≠ West
Karan BoseWest1,200,000kept
Notice the first condition compares text (with quotes: 'West') and the second compares a number (without quotes: 500000). Both are in the same WHERE clause.

Part 3
Aggregate Functions
These are built-in calculators. They take many rows and return one number — a count, a total, an average, etc.
Aggregate functions are SQL keywords (blue words). They are always spelled exactly the same: COUNT, SUM, AVG, MIN, MAX.
But the column name you put inside the brackets — that depends on YOUR data.
The pattern
SELECT FUNCTION_NAME(any_column) FROM table_name;
any_column means any column from your table — not just columns with "name" in them. It could be deal_value, region, hire_date, salary — literally any column.
Immediate example
SELECT COUNT(lead_name) FROM leads;
Returns → 4 (counts all lead names)
SELECT SUM(deal_value) FROM leads;
Returns → 4,150,000 (adds up all deals)
Aggregate functions work with just SELECT + FROM. You don't need WHERE or GROUP BY to use them — those are separate, optional features.
Which function works with which type of data?

Columns hold different types of data — numbers, text, dates. Not every function works with every type.

FunctionNumbers
salary, price, age		Text
name, city, status		Dates
hire_date, order_date

SUM( )YesNoNo
AVG( )YesNoNo

COUNT works on anything — it just counts rows. It doesn't care if the column holds names, numbers, or dates.
SUM and AVG only work on numbers. You can't add up city names or average a list of dates. If you try, SQL gives an error.
MIN and MAX work on anything. For numbers: smallest/largest. For text: alphabetically first/last ('Apple' < 'Banana'). For dates: earliest/latest.

Same MAX. Different column, different table.

Naming your result with AS

When you use an aggregate function, SQL doesn't know what to call the result column. AS lets you give it a name — any name you want.

Part 4
GROUP BY
Takes your rows, puts them in groups based on a column, then runs aggregate functions on each group separately.

In Part 3, you learned COUNT(*) gives you one number for the whole table. But what if your boss asks: "How many leads do we have in each region?"

That's GROUP BY. It splits your data into piles — one pile per region, or department, or category — and then counts (or sums, or averages) each pile separately.

Without GROUP BY

One answer for everybody

SELECT COUNT(*) FROM leads; → 4
With GROUP BY

One answer per group

SELECT region, COUNT(*) FROM leads GROUP BY region; → West: 2, North: 1, South: 1
A Pivot Table in Excel. You drag "Region" to the Rows area, and "Deal Value" to the Values area with "Sum". GROUP BY is the same — group by region, sum the deal values.
The pattern
SELECT grouping_column, FUNCTION(column) AS name FROM table GROUP BY grouping_column;
What each word does
GROUP BY Tells SQL: "put all rows with the same value in this column into one group". Then each function runs once per group.
AS name Gives your calculated column a label. Without it, the column header would just say COUNT(*) instead of something readable.
GROUP BY doesn't need WHERE. You can group all rows without filtering first. WHERE is only added when you want to exclude some rows before grouping.
GROUP BY doesn't change your original data. Your table stays exactly the same. GROUP BY only changes the results you see — like applying a filter in Excel doesn't delete your rows.
Example — watch it happen step by step
SELECT region, COUNT(*) AS total_leads, SUM(deal_value) AS revenue, AVG(deal_value) AS avg_deal FROM leads GROUP BY region;

The #1 rule beginners trip on

When you use GROUP BY, each group collapses into one row. So every column in your SELECT must either be:

Allowed
The column you're grouping by — because each group only has one value for it (e.g., "West")
Allowed
Inside a function like COUNT, SUM, AVG — because the function calculates a single answer from multiple rows

But what about a regular column like lead_name? Watch what happens:

The "West" group has these lead names:
Riya Mehta Karan Bose
The group becomes one row — but which name should SQL show? Riya? Karan? It can't pick, so it gives up and shows an error.
This breaks
SELECT region, lead_name, COUNT(*) FROM leads GROUP BY region;

lead_name isn't grouped and isn't in a function. SQL doesn't know which name to pick.

This works
SELECT region, COUNT(*), SUM(deal_value) FROM leads GROUP BY region;

Everything is either the group column or inside a function.

"I need to add WHERE before GROUP BY, otherwise it won't work"
WHERE and GROUP BY are completely independent. WHERE filters before grouping (like removing rows from your spreadsheet before making a pivot table). You only add WHERE when you want to exclude certain rows. Most GROUP BY queries don't have WHERE at all.
"GROUP BY changes my original table"
GROUP BY only affects what your results look like. Your table still has every single row. Think of it like putting sticky notes on your rows — "you're West, you're North" — counting each pile, and giving you a summary. The original rows are untouched.
"I can only group by one column"
You can group by multiple columns! GROUP BY region, status creates a group for each unique combination: "West + Won", "West + Lost", "North + Won", etc. It's like a pivot table with two row fields.
Same idea, different names
SELECT department, COUNT(*) AS headcount, AVG(salary) AS avg_salary FROM employees GROUP BY department;
Different data, same shape. Group by department instead of region. Count employees instead of leads.
Your manager asks: "How many orders did we get from each city last month?" That's GROUP BY. You group by city, count the orders, and maybe add a WHERE to limit it to last month. In a BA role, nearly every report you build will use GROUP BY — "sales by region", "users by plan type", "tickets by priority". It's the most common pattern in business reporting.

ORDER BY + LIMIT
Sort your results and control how many rows to show.
The pattern
SELECT columns FROM table ORDER BY column DESC LIMIT number;
What each word does
ORDER BY Sorts the result. Must come near the end of your query (after WHERE and GROUP BY).
DESC Largest first (descending). Think: Down.
ASC Smallest first (ascending). Think: Arrow up. This is the default — if you don't write anything, SQL assumes ASC.
LIMIT Only return this many rows. "Give me the top 3" = LIMIT 3.
-- text Two dashes start a comment. SQL ignores everything after -- on that line. It's just a note to yourself.
Clicking a column header in Excel to sort A→Z or Z→A, then deleting all rows below a certain point. ORDER BY sorts, LIMIT cuts.
Each keyword works on its own
ORDER BY and LIMIT are independent. You can use one without the other, and neither requires WHERE or GROUP BY. They're separate tools you add only when needed.
Three different queries, each valid
Just LIMIT
SELECT * FROM leads LIMIT 2;
Just give me any 2 rows
Just ORDER BY
SELECT * FROM leads ORDER BY deal_value;
Sort all rows, show all of them
Both together
SELECT * FROM leads ORDER BY deal_value DESC LIMIT 3;
Sort + top 3 = "Top N" list
Full example
SELECT lead_name, deal_value FROM leads ORDER BY deal_value DESC LIMIT 3; -- top 3 deals only
#lead_namedeal_value
1Arjun Nair1,500,000
2Karan Bose1,200,000
3Riya Mehta800,000
Notice: no WHERE here. We didn't filter anything — we just sorted and took the top 3. You add WHERE only if you want to exclude rows first.
Check yourself
Will this query work? SELECT * FROM products LIMIT 5;
Correct! LIMIT works on its own. It just says "give me this many rows" — no WHERE or ORDER BY required.
LIMIT is independent! It doesn't need WHERE or ORDER BY. SELECT * FROM products LIMIT 5; is perfectly valid — gives you any 5 rows.

JOIN — Combining two tables
When your data lives in two separate tables, JOIN connects them using a column they share.
First — why is data in separate tables?

Imagine you had ONE giant spreadsheet with leads AND campaign info. Every time Riya's row appears, you'd also have to write "Q1 LinkedIn", "LinkedIn" — and if that campaign name changes, you'd have to update it in every single row.

So databases split things into separate tables: leads in one, campaigns in another. Each has its own job. They're connected by a shared ID — a number or code that appears in both tables, like a reference number.

leads table

The blue IDs are the link between the two tables. campaign_id in leads matches id in campaigns. This is how SQL knows which rows belong together. Think of it like a reference number on a receipt — it connects the receipt to the order.
How to spot the link column

When you look at a table, the link column usually follows a pattern: othertable_id. For example:

In the leads table
campaign_id — "campaign" tells you it points to the campaigns table. "id" tells you it's a reference number.
In the orders table
customer_id — "customer" tells you it points to the customers table. Same pattern.
The naming pattern: If a column is called something_id, it almost always links to the id column in the something table. customer_id → customers.id. product_id → products.id. Learn this pattern and JOIN becomes much easier.
The pattern
SELECT columns_from_either_table FROM first_table AS shortname1 JOIN second_table AS shortname2 ON shortname1.link_column = shortname2.id;
What each word does
FROM Your starting table — the one with the main data you care about.
JOIN The second table you want to bring in. "Also get data from this table."
ON The matching rule. "Connect rows where this column equals that column." This is where you put the link.
AS l / AS c Short nicknames so you don't type the full table name every time. l = leads, c = campaigns.
l.column The dot means "from this table". l.lead_name = "the lead_name column from the leads table".
VLOOKUP in Excel. You take campaign_id from the leads sheet, look it up in the campaigns sheet, and bring over the campaign_name and channel columns. JOIN does this automatically for every row.
Example — watch SQL match the rows
SELECT l.lead_name, c.campaign_name, c.channel, l.deal_value FROM leads AS l JOIN campaigns AS c ON l.campaign_id = c.id;

What if a row doesn't have a match?

Say there's a campaign C4 — "Summer Sale" in the campaigns table, but no lead has campaign_id = C4. What happens?

??? — no lead has C4
✗
C4 = Summer Sale — no match, dropped

JOIN (aka INNER JOIN) drops unmatched rows entirely. If a campaign has no leads, it won't appear in your results. If a lead has no matching campaign, same thing — gone.

JOIN = "only show me rows that exist in BOTH tables." Think of it like a Venn diagram — you only get the overlap. This is the most common type of join, and the safest to start with. There are other types (like LEFT JOIN) that keep unmatched rows, but those are for later.
The dot (.) — your biggest new friend

When you work with two tables, they might both have a column called id. If you just write id, SQL asks: "Which table's id? I see two!" The dot solves this:

Ambiguous
SELECT id, lead_name ...

Which table's id? Leads or campaigns? SQL doesn't know.

Clear
SELECT l.id, l.lead_name ...

l.id = "the id from the leads table". No confusion.

Good habit: When using JOIN, always use the dot prefix on every column, even if the name is unique. It makes your query easier to read and prevents errors if someone adds a column with the same name later.
"JOIN creates a new permanent table"
JOIN only creates a temporary combined view in your results. Your original tables are completely unchanged. It's like placing two Excel sheets side by side, drawing lines between matching rows, and reading across — the sheets themselves don't change.
"I need to memorize which table goes in FROM and which goes in JOIN"
It doesn't matter! FROM leads JOIN campaigns gives the same result as FROM campaigns JOIN leads (as long as the ON clause matches). Most people put their "main" table in FROM and the "lookup" table in JOIN, but SQL doesn't care about the order.
"The ON clause looks scary — I don't know what goes there"
There's a simple trick: find the something_id column. If the leads table has campaign_id, then your ON is: leads.campaign_id = campaigns.id. The _id column always points to the id of the other table. That's the pattern for 90% of joins you'll ever write.
Same idea, different names
SELECT o.order_id, c.customer_name, o.amount FROM orders AS o JOIN customers AS c ON o.customer_id = c.id;
Orders + Customers instead of Leads + Campaigns. The pattern is identical: find the _id column, match it to the other table's id.
Your manager asks: "I need a list of all orders with the customer name and email attached." The orders table has customer_id but not the name. The customers table has the name but not the order details. JOIN brings them together. In real BA work, almost no useful report comes from just one table — you're always combining orders with customers, products with categories, employees with departments. JOIN is how you connect the dots.

The question we want to answer:
"For deals over 7 lakhs, show me total leads and revenue by marketing channel — highest revenue first."
SELECT c.channel, COUNT(*) AS leads, SUM(l.deal_value) AS total_revenue FROM leads AS l JOIN campaigns AS c ON l.campaign_id = c.id WHERE l.deal_value > 700000 GROUP BY c.channel ORDER BY total_revenue DESC;
